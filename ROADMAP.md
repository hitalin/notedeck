# NoteDeck Roadmap

## ポジショニング

NoteDeck は Misskey とそのフォークに対応した**マルチプラットフォーム対応デッキクライアント**である。

- **ターゲット**: 複数サーバーのタイムラインを効率よく閲覧するユーザー
- **対象プラットフォーム**: Windows / macOS / Linux / Android
- **ゴール**: 日常操作の 80% を NoteDeck 内で完結させ、Web UI を開く頻度を最小化する

### プラットフォームごとの体験

- **デスクトップ**: デッキ UI（複数カラム同時表示）、キーボード操作、ローカル検索、AI 統合
- **モバイル**: シングルカラム + タブ切替による軽快な閲覧体験。デッキの全カラムにアクセス可能
- **AI 機能はデスクトップ専用**: ローカル LLM（Ollama）や常駐 + ローカル DB を前提とするため

### なぜ NoteDeck か — フォーク対応

Misskey のフォーク（Sharkey・CherryPick・Firefish・Iceshrimp 等）はそれぞれ独自機能を持つが、
モバイルクライアントは本家 API にしか追従できず、フォーク独自機能を使うには結局 PWA に戻るしかない。

NoteDeck は adapter パターンでフォークごとの差異を吸収し、**どのフォークでも固有機能が使えるデスクトップクライアント**を目指す。
PWA の機能網羅性と、ネイティブアプリのデスクトップ体験（デッキ UI・キーボード操作・ローカル検索）を両立する。

## 競合分析・差別化戦略

### デスクトップクライアント市場

NoteDeck は Tauri（Vue 3 + Rust）で構築された**唯一のデスクトップ特化 Misskey クライアント**。
Flutter 製の Aria がマルチプラットフォーム対応で存在するが、デスクトップネイティブの Tauri 製は NoteDeck だけ。

| | NoteDeck (Tauri) | Aria (Flutter) | Misskey Web (PWA) |
|---|---|---|---|
| デッキ UI（マルチカラム） | ✅ 22 種カラム | ❌ | ✅（デッキモード） |
| ローカル DB + FTS5 全文検索 | ✅ | ❌ | ❌ |
| localhost HTTP API（外部連携口） | ✅ port 19820 | ❌ | ❌ |
| システムトレイ常駐 | ✅ | ❌ | ❌ |
| グローバルホットキー（非フォーカス時） | ✅ | ❌ | ❌ |
| OS キーチェーン連携 | ✅ | △ 限定的 | ❌ |
| 自動アップデート（署名検証付き） | ✅ | △ ストア経由 | ❌ |
| マルチウィンドウ・マルチモニター | ✅ 計画中 | ❌ | ❌ |
| 外部プロセス起動（Ollama 等） | ✅ | ❌ | ❌ |
| ファイルシステム直接アクセス | ✅ | △ 制限付き | ❌ |
| クリップボード深層統合 | ✅ 計画中 | △ | ❌ |
| 24 時間常駐自動化 | ✅ 計画中 | ❌ | ❌ |
| AiScript プラグイン（Misskey 互換） | ✅ | ❌ | ✅（本家） |
| モバイル対応 | △ Android のみ | ✅ 全プラットフォーム | ✅ ブラウザ |

### 棲み分け戦略

Aria とは**競合ではなく棲み分け**。デスクトップでは NoteDeck、モバイルでは Aria が最適解。

```
Aria の戦略:     広く浅く → 全プラットフォームで 70 点
NoteDeck の戦略:  狭く深く → デスクトップで 150 点
```

NoteDeck の差別化は**デスクトップネイティブでしか原理的に不可能な機能**に集中する。
Web UI / Flutter アプリが「頑張れば追いつく」ではなく「アーキテクチャ的に不可能」な領域が堀になる。

### localhost API — 最大の構造的優位

port 19820 の HTTP API は、Tauri の Rust バックエンド（Axum）だからこそ自然に載る。
Flutter / Web には原理的に不可能。この API を**外部ツール連携の正式プロトコル**に育てることで、
NoteDeck を Misskey エコシステムのハブにする。

```
連携例:
├── Alfred / Raycast ワークフロー → ランチャーから直接投稿
├── Obsidian プラグイン → サイドバーに Misskey の関連ノートを表示
├── VS Code 拡張 → コード書きながらサイドバーで TL 監視
├── シェルスクリプト / cron → 定期キーワード検索 → 結果を通知
├── Hammerspoon / AutoHotKey → 選択テキストを即投稿
└── StreamDeck → 物理ボタンで投稿・TL 切替・ミュート
```

## 設計原則

- Apple 式直感 UI: 設定項目を増やさず、触ればわかる操作で完結
- 機能網羅より体験品質: 少ない機能を心地よく使えることを優先
- Web UI へのリンク導線: 非対応機能はブラウザで開けるようにする

---

## 構造的に対応しない領域

低頻度・管理系の操作は Web UI の責務とし、実装しない。
**ただし、ユーザーが迷わず Web UI へ到達できるリンク導線を整備する。**

> **注**: この判断基準は**一般ユーザー**に対するもの。
> サーバー管理者・フォーク開発者向けには Phase 11 で管理・開発機能を提供する。
> 管理者にとってモデレーション・サーバー監視は「低頻度」ではなく「毎日の業務」であり、
> 一般ユーザーとはペルソナが異なる。

| 領域 | 理由 | リンク導線 |
|---|---|---|
| 管理画面 | 一般ユーザーには不要。管理者向けは Phase 11 で対応 | ✅ アカウントメニュー → Admin |
| プロフィール編集 | 年に数回の操作。Web UI リンクで十分 | ✅ プロフィールバナー → 編集ボタン |
| リスト管理 | 作成・編集は Web UI で。カラム表示のみ対応 | ✅ カラムヘッダー → 外部リンク |
| アンテナ管理 | 作成・編集は Web UI で。カラム表示のみ対応 | ✅ カラムヘッダー → 外部リンク |
| チャンネル管理 | 閲覧は対応。作成・編集は Web UI で | ✅ カラムヘッダー → 外部リンク |
| ミュート・ブロック | 設定の下位項目。一覧管理は Web UI で | ✅ アカウントメニュー → Settings |
| フォロー管理画面 | 一覧表示・整理は Web UI で。フォロー操作のみ対応 | ✅ プロフィール → Following / Followers |
| 設定画面の網羅 | Misskey 本家の膨大な設定項目は Apple 式の思想と矛盾 | ✅ アカウントメニュー → Settings |
| ノート詳細 | アプリ内で会話チェーン・タブ表示済み。Web UI リンクは補助導線 | ✅ ノートメニュー → Web UIで開く |
| 他人のプロフィール | アプリ内で閲覧・ノート一覧・フォロー操作済み。Web UI リンクは補助導線 | ✅ プロフィールバナー → 外部リンクボタン |
| Misskey Games | 物理演算ゲーム・リアルタイム対戦の移植は過大。Web UI で遊べば十分 | ✅ アカウントメニュー → Misskey Games |

**判断基準**: 月 1 回以下の操作 / 作成・管理系 / Web UI リンクで代替可能（一般ユーザー基準）

## Core: 基盤機能

Phase を横断する土台。デッキ UI・レンダリング・マルチアカウント・デスクトップ統合の基礎。

### デッキ UI

- [x] **カラム管理** — 追加・削除・並べ替え（ドラッグ＆ドロップ）、レイアウト localStorage 永続化
- [x] **カラムリサイズ** — ドラッグで 280px〜600px に自由変更、幅を localStorage に永続化
- [x] **ナビバー構造** — 折りたたみ/展開（68px↔250px）、ドラッグリサイズ、120px 以下で自動折りたたみ
- [x] **フローティングウィンドウ** — ノート詳細・ユーザープロフィール・ログインをウィンドウで表示。
  ドラッグ移動、最小化/最大化、Z-index 自動管理、同一対象の重複排除、サーバーテーマ反映
- [x] **カスタムタイトルバー** — 最小化/最大化/閉じる、サイドバー切替、モバイルサイズ切替
- [x] **カラムフィルター** — Renote 除外・リプライ除外・ファイル付きのみ・Bot 除外（timelineFilter）

構造的に対応しない領域への Web UI リンク。ユーザーが迷わず Web UI へ到達できるようにする。

- [x] **アカウントメニュー → Admin** — 管理者のみ表示、外部ブラウザで管理画面を開く

**カラムで再現できる機能はナビバーに置かない。** カラムを超えた操作のみをナビバーに配置する。

## Phase 2: 唯一無二の価値 — デスクトップアプリだからこそ

Phase 1 で蓄積されたローカルキャッシュの上に、Web UI では原理的に不可能な機能を載せる。

- [x] **ローカル全文検索基盤** — SQLite FTS5（trigram トークナイザー）による日本語対応検索。
  ストリーミング受信・API 取得時にノートを自動キャッシュし、検索カラムで即座にローカル検索。
  サーバー検索との統合表示（重複排除・マージ）も実装済み
- [x] **キャッシュファースト表示** — 全カラムでローカルキャッシュを先読みし即座に表示。
  サーバーからは差分のみ取得（sinceId）。オフライン時もキャッシュ済み TL を表示継続
- [x] **横断検索** — 複数サーバー × 全カラムのノートを統合検索。
  全アカウントのローカルキャッシュ＋サーバー検索を並列実行し、結果をマージ表示。
  サーバー A のリスト + サーバー B のアンテナ + サーバー C の HTL をまたいだ検索は NoteDeck だけの体験

**Phase 2 完了** — SQLite FTS5 + キャッシュファースト表示 + 差分取得 + 横断検索が揃い、
ローカルファーストの検索体験が完成。

## Phase 3: Hackable — VSCode の世界観を Misskey に

NoteDeck を拡張可能にする 3 本柱。VSCode がエディタの世界で実現したモデルを、
Misskey エコシステムの言語（AiScript）とプロトコルで再構築する。

**3 つの柱が同じコマンドシステムに収斂する:**
- **人間** → コマンドパレットから実行
- **AI** → HTTP API から実行
- **AiScript プラグイン** → コマンドを登録・実行

### 柱 1: コマンドパレット（人間のインターフェース）

VSCode の Ctrl+Shift+P に相当。全操作がコマンドとして登録され、検索・実行できる。

- [x] コマンドパレット基盤（Ctrl+K で起動、fuzzy 検索）
- [x] ショートカットキー連携（コマンドにキーバインドを紐づけ）
- [x] **キーバインドカスタマイズ** — `useKeybindsStore` でデフォルト＋ユーザー上書きを管理。
  VSCode の `keybindings.json` と同じ思想。設定 UI からリマップ可能。
  カラム間移動・クイックリアクション等の全キーバインドを統合管理

### 柱 2: HTTP API（AI・外部ツールのインターフェース）

VSCode の Extension API + Language Server に相当。
Claude Code や AI エージェントが NoteDeck を操作できる外部 API。

- [x] HTTP サーバー基盤（localhost で起動）
- [x] **Bearer トークン認証** — 起動時にランダムトークンを生成しファイルに書き出し。
  Jupyter Notebook 方式で同一マシンの認可済みプロセスのみアクセス可能
- [x] **コマンド実行 API** — 登録済みコマンドを HTTP 経由で実行。
  コマンドパレットで人間ができることは全て API からもできる
- [x] **状態取得 API** — カラム一覧、アクティブカラム、ノート取得等の読み取り操作
- [x] **イベントストリーム** — SSE で状態変化を購読。
  AI が「新着通知があったら要約して」等のリアクティブ処理を実現

### 柱 3: AiScript プラグイン（Misskey ネイティブの拡張言語）

VSCode の TypeScript Extensions に相当するが、言語は **AiScript**。
Misskey ユーザーが既に馴染んでいるサンドボックス言語で、安全に拡張機能を書ける。

- **配布が容易** — テキストベースでコピペ共有可能。Misskey 本家と同じインストール体験

## Phase 6: AI 統合 — 2 つのドア、同じ家

NoteDeck に AI を統合する。ただし AI を「持つ」のではなく「受け入れる」プラットフォームとして。
IT に詳しくないクリエイターも、ハッカーも、同じ AI 基盤の恩恵を受けられる設計にする。

### なぜ AI 機能はデスクトップ限定か

- **ローカル LLM（Ollama）がコア**: スマホではローカル LLM を動かせない
- **常駐 + ローカル DB + AI の三位一体**: スマホにはローカル DB の蓄積も常駐も弱い
- **API キー管理の UX**: スマホユーザーに API キーを入力させるのは Apple 式思想に反する
- **中途半端な AI より、ない方が誠実**: クラウド API 経由の遅いレスポンスを返すくらいなら、ない方が UX として正しい
- **モバイル版では AI 関連の UI（ナビバーボタン・カラム追加・設定）を非表示にする**

```
クリエイター（非 IT）              ハッカー（IT）
      │                              │
  「AI アシスタント」              notecli CLI / HTTP API
   ボタンを押すだけ               Claude Code・OpenClaw から直接操作
      │                              │
      └──────────┬───────────────────┘
                 │
         NoteDeck AI 基盤
         (同じ内部 API)
```

**既存の土台:**
- Phase 3 の HTTP API + SSE + コマンドシステムが完成済み
- Phase 4 の notecli CLI（`--json` 出力）で AI エージェントから全操作可能
- 人間がコマンドパレットでできることは、CLI / API 経由で AI にもできる

### なぜ新プロトコル（MCP 等）を追加しないか

notecli CLI + notedeck HTTP API で AI が必要とする操作は全てカバー済み。

```bash
# notecli: データ操作（CLI の世界）— 任意の AI が Bash 経由で利用可能
notecli post "投稿内容"
notecli search "キーワード" --json | jq '.[] | .text'
notecli tl home --limit 50 --json

# notedeck HTTP API: GUI 状態操作（ステートフルな世界）
curl localhost:19820/api/deck/columns
curl localhost:19820/api/commands/compose/execute
curl localhost:19820/api/events  # SSE でリアルタイム購読
```

| | CLI (notecli) | HTTP API (notedeck) | MCP |
|---|---|---|---|
| 既に実装済み | ✅ | ✅ | ❌ |
| Claude Code 対応 | ✅ Bash 経由 | ✅ curl 経由 | ✅ ネイティブ |
| 任意の AI 対応 | ✅ シェルがあれば | ✅ HTTP があれば | △ MCP 対応のみ |
| Unix composable | ✅ パイプ・jq | △ | ❌ |
| 追加実装コスト | 0 | 0 | 中〜大 |

**既にある武器を磨く方が、新しいプロトコル層を追加するより正しい。**
MCP は需要が出た時点で CLI / API の薄いラッパーとして追加すれば十分。

### 6-1. Ollama 統合 + AI カラム（クリエイター向け）

ローカル LLM（Ollama）をファーストクラスでサポートし、ボタン 1 つで AI 機能を提供する。
Ollama はローカルファースト思想と一致する: 無料・データが PC から出ない・オフライン動作。

| | クラウド AI | Ollama |
|---|---|---|
| コスト | 従量課金 | 無料 |
| プライバシー | ノートが外部送信される | PC から出ない |
| オフライン | 使えない | 使える |
| NoteDeck の思想 | ❌ サーバー依存 | ✅ ローカルファースト |

- [ ] **AI プロバイダー設定** — 初回セットアップ 1 画面で完結。
  Ollama 自動検出（起動中なら即接続）、OpenAI / Anthropic / OpenAI 互換エンドポイント対応
- [ ] **ローカル DB コンテキスト** — 全ノートを文脈として渡せる（Web サービスには不可能な量）

**UI/UX 戦略 — 新しい UI 要素を発明しない:**

既存の 3 パターン（インライン・ウィンドウ・カラム）に AI を乗せる。
右サイドバー等の固定パネルは作らない（デッキ UI のカラム幅を圧迫するため）。

```
┌─────┬──────────┬──────────┬──────────┐
│     │ Timeline │ Notif    │ AI Chat  │  ← カラムとして追加（ユーザーが選ぶ）
│ Nav │          │          │          │
│     │          │          │          │
│  AI │          │          │          │  ← ナビバーに AI ボタン
│     │          │          │          │     （ウィンドウで開く）
└─────┴──────────┴──────────┴──────────┘
```

| レイヤー | UI | いつ使う |
|---|---|---|
| インライン | ノートメニュー → 翻訳結果をノート内に表示 | 1 ノートに対する操作 |
| ウィンドウ | ナビバーの AI ボタン → フローティングウィンドウ | クイック質問・TL 要約 |
| カラム | カラム追加 → AI チャット | じっくり会話 |

- [ ] **AI アシスタントウィンドウ** — ナビバーに AI ボタンを配置。

  クイック質問・TL 要約など一時的な操作用
- [ ] **AI チャットカラム** — カラム追加ダイアログから追加可能。
  じっくり AI と会話する場合のインターフェース。他カラムと同じくリサイズ・削除可能
- [ ] **ワンボタン AI 機能** — プロンプトを書かせない UI:
  - カラムヘッダーの「要約」ボタンで TL まとめ
  - ノートメニュー →「翻訳」で即翻訳（結果はノート内にインライン表示）
  - 検索カラムで自然言語入力（「この人との過去のやり取り」等）

### 6-2. AI × AiScript — 自己拡張するアプリ

AI が AiScript プラグインを生成し、NoteDeck 自身を拡張する。

### 6-3. AI 連携ユースケース

## Phase 7: マルチウィンドウ — デスクトップネイティブの真価

パワーユーザーはマルチモニター環境で作業している。
Web UI / PWA には**原理的に不可能な**デスクトップネイティブの強みを解放する。

### なぜこれが重要か

- Misskey 廃人の多くはマルチモニター環境。モニター 1 に TL、モニター 2 に通知+検索は自然な要求
- Tauri のマルチウィンドウ API で技術的に実現可能
- デッキプロフィール（Phase 5）との統合で、ウィンドウレイアウトごと保存・復元できる
- これは PWA / Web UI との**不可逆な差別化**であり、「デスクトップアプリである理由」を強化する

### 7-1. マルチウィンドウ

- [ ] **デッキのウィンドウ分離** — カラムグループを別ウィンドウに分離して表示。
  モニター 1 に TL 系カラム、モニター 2 に通知 + 検索 + AI カラム等の配置が可能
- [ ] **ウィンドウ間カラム移動** — ドラッグ＆ドロップでカラムをウィンドウ間で移動
- [ ] **ウィンドウレイアウト永続化** — デッキプロフィールにウィンドウの位置・サイズ・カラム構成を含めて保存。
  `Alt+1`〜`9` で切替時にウィンドウレイアウトごと復元

### 7-2. マルチモニター最適化

- [ ] **モニター記憶** — ウィンドウがどのモニターに配置されていたかを記憶し、次回起動時に復元
- [ ] **モニター切断時のフォールバック** — モニターが外れた場合、カラムをメインウィンドウに自動マージ

### 7-3. ウィンドウモード拡張

Web UI / Flutter には不可能なデスクトップネイティブのウィンドウ操作。

- [ ] **PiP（ピクチャ・イン・ピクチャ）モード** — NoteDeck を最小ウィンドウにして TL だけ流す。
  常に最前面表示（Always on Top）でゲーム中・作業中に TL を横目で監視
- [ ] **カラム分離ウィンドウ** — カラムをドラッグして本体から切り離し、独立ウィンドウに。
  モニター 1: メインデッキ、モニター 2: 通知カラム単体のような使い方
- [ ] **ウィンドウ透過・背景ぼかし** — ウィンドウの透過率を設定可能。
  macOS の vibrancy / Windows の Mica 素材対応で、デスクトップ壁紙が透けて見える半透明 TL
- [ ] **ウィンドウスナップ統合** — OS のスナップ機能（Win+矢印 / macOS Split View）に適切に対応。
  画面半分 = NoteDeck、半分 = ブラウザのような使い方を快適に

## Phase 8: AiScript エコシステム — 拡張機能プラットフォームへ

Phase 3 で AiScript プラグイン基盤は完成したが、**発見・共有・配布のインフラ**がない。
VSCode の Extension Marketplace が VSCode を支配的にしたように、
エコシステムの厚みがロックインを生む。

### NoteDeck AiScript の差別化

Misskey 本家のプラグインは Web UI 内のサンドボックスに閉じている。
NoteDeck のプラグインは以下にアクセスできる唯一の AiScript 実行環境:

| 能力 | Misskey 本家 | NoteDeck |
|---|---|---|
| ノート操作（Mk:api） | ✅ | ✅ |
| UI 拡張（Plugin:*） | ✅ | ✅ |
| ローカル DB 検索 | ❌ | ✅ Nd:db |
| 外部 API コール | ❌ | ✅ Nd:http |
| コマンドパレット統合 | ❌ | ✅ Nd:register_command |
| カラム操作 | ❌ | ✅ Nd:columns / Nd:addColumn |
| イベント購読 | ❌ | ✅ Nd:on |
| 自動化ルール | ❌ | ✅ Nd:automation |

**「Misskey で外部 API を叩ける唯一のプラットフォーム」** という立ち位置を確立する。

### 8-1. Nd:* API 拡充

Phase 3 で実装済みの `Nd:register_command` / `Nd:columns` / `Nd:addColumn` / `Nd:removeColumn` に加え、
プラグインの可能性を広げる API を追加。

- [ ] **Nd:db.query** — ローカル DB（SQLite FTS5）への読み取りアクセス。
  プラグインが蓄積された全ノートを検索・集計できる
- [ ] **Nd:db.stats** — ノート数・ユーザー数・期間等の統計情報取得
- [ ] **Nd:http** — 外部 HTTP API コール（GET/POST）。
  翻訳 API・天気 API・RSS フィード等との連携が可能。CORS 制約なし（デスクトップネイティブの強み）
- [ ] **Nd:on** — イベント購読 API。ストリーミングイベント（新着ノート・通知・フォロー等）を
  プラグインがリアクティブに受け取れる
- [ ] **Nd:storage** — プラグイン専用のローカルストレージ（Mk:save/load の拡張版）。
  名前空間分離・容量制限付き
- [ ] **Nd:ui.notify** — プラグインからデスクトップ通知を送信
- [ ] **Nd:accounts** — ログイン済みアカウント一覧の取得（トークンは非公開）

### 8-2. プラグインストア

プラグインの発見・インストール・更新を一元管理する仕組み。

- [ ] **プラグインギャラリー UI** — アプリ内でプラグインを検索・プレビュー・ワンクリックインストール。
  カテゴリ分類（フィルター / 翻訳 / UI カスタマイズ / 自動化 / 連携 等）
- [ ] **プラグインレジストリ** — GitHub リポジトリベースのプラグインインデックス（JSON）。
  メタデータ（名前・説明・作者・バージョン・対応 NoteDeck バージョン）を集約
- [ ] **プラグイン更新通知** — インストール済みプラグインの新バージョンを検知し通知
- [ ] **プラグインレビュー・スター** — コミュニティによる品質シグナル

### 8-3. プラグインテンプレート・開発支援

プラグイン作者を増やすための開発体験の整備。

- [ ] **テンプレートギャラリー** — 「TL フィルター」「外部 API 連携」「カスタムカラム」
  「通知カスタマイズ」等のユースケース別テンプレートから開発を開始できる
- [ ] **プラグイン開発ドキュメント** — Nd:* API リファレンス、チュートリアル、サンプル集
- [ ] **ホットリロード** — AiScript エディタカラムで編集 → 即座にプラグインに反映。
  保存するたびにアプリ再起動不要

## Phase 9: AiScript 自動化 — 常駐アプリが動かすワークフロー

常駐デスクトップアプリ + ローカル DB + AiScript の三位一体で、
イベント駆動の自動化ルールを実現する。IFTTT / Zapier の Misskey 版。

### なぜ NoteDeck でしかできないか

- **常駐** — Web UI はブラウザを閉じたら終わり。NoteDeck は 24 時間動き続ける
- **イベントストリーム** — WebSocket で全イベントをリアルタイム受信済み（Phase 1）
- **ローカル DB** — 蓄積データに対する条件判定が可能（「今日 N 件以上ノートしたら」等）
- **AiScript** — ルールのロジックを安全なサンドボックスで記述
- **Nd:http** — 外部サービスとの連携（Webhook 送信・API コール等）

### 9-1. 自動化エンジン

- [ ] **トリガー → アクション ルールエンジン** — AiScript で記述するイベント駆動ルール。
  `Nd:on("note", ...)` でノート受信をトリガーに、条件判定 → アクション実行
- [ ] **組み込みトリガー**:
  - 新着ノート（キーワード・ユーザー・カラム単位）
  - 通知（タイプ別: リアクション・フォロー・メンション等）
  - フォロワー数変化
  - 時刻（cron 的な定期実行）
- [ ] **組み込みアクション**:
  - デスクトップ通知（カスタムサウンド・優先度指定）
  - 自動ブックマーク / 自動クリップ
  - 外部 Webhook 送信（Discord / Slack 通知等）
  - ノートのローカルタグ付け（Phase 10 と連携）
- [ ] **自動化ルール管理 UI** — ルール一覧・有効/無効切替・実行ログ閲覧

### 9-2. ユースケース例

```
# 特定ユーザーのノートを自動ブックマーク
Nd:on("note", @(note) {
  if (note.user.username == "syuilo") {
    Mk:api("notes/favorites/create", { noteId: note.id })
    Nd:ui.notify("syuilo さんがノートしました")
  }
})

# 特定キーワードを含むノートを Discord に転送
Nd:on("note", @(note) {
  if (note.text != null && note.text.incl("NoteDeck")) {
    Nd:http("POST", "https://discord.com/api/webhooks/...", {
      content: `{note.user.name}: {note.text}`
    })
  }
})
```

## Phase 10: PKM — ローカル DB をナレッジベースに昇格

Phase 5-4 の「ノートキャッシュ永久保存」を発展させ、
ローカル DB を**個人ナレッジベース（Personal Knowledge Management）**に昇格する。

思想：**サーバーが消えても、あなたのタイムラインはここにある。**

### なぜこれが NoteDeck の自然な進化か

- ローカル DB に全ノートが蓄積される基盤は既にある（Phase 2・5-4）
- FTS5 全文検索も実装済み（Phase 2）
- あとは「キャッシュ」を「知識」として扱う UI とメタデータを載せるだけ

### 10-1. ローカルメタデータ

サーバーのブックマーク / クリップとは独立した、ローカル専用の整理機能。
サーバー側にリクエストを送らないので、サーバー負荷ゼロ・オフラインで完結。

- [ ] **ローカルタグ** — ノートに自由なタグを付与（サーバーに送信しない）。
  「技術メモ」「あとで読む」「名言」等の個人分類
- [ ] **ローカルスター** — ブックマークとは別の「お気に入り度」。★1〜5 の重要度ランク
- [ ] **ノートコレクション** — タグをまとめたスマートフォルダ。条件指定で自動収集も可能
  （例: 「タグ:技術メモ AND ★3以上」）
- [ ] **ノートメモ** — ノートに自分だけのメモ・コメントを追記。
  「この方法試したらうまくいった」等の文脈を後から残せる

### 10-2. ナレッジブラウザ

蓄積されたノートを「タイムライン」ではなく「知識」として閲覧する新しいビュー。

- [ ] **コレクションカラム** — ローカルタグ / コレクション単位でノートを表示するカラム種別
- [ ] **タグクラウド** — 使用頻度の高いタグを可視化。タグクリックでフィルター
- [ ] **タイムライン統計** — 蓄積ノート数・期間・サーバー別内訳等のダッシュボード

### 10-3. エクスポート・相互運用

ローカル DB のデータをユーザーの手元に完全に解放する。

- [ ] **Markdown エクスポート** — ノート・コレクション単位で Markdown ファイルに書き出し。
  Obsidian / Notion / HackMD 等との連携
- [ ] **JSON エクスポート** — 機械可読形式での全データエクスポート。バックアップ・移行用
- [ ] **コレクション共有** — コレクションを JSON でエクスポート → 他の NoteDeck ユーザーにインポート

### AI との合流点（Phase 6 連携）

PKM と AI 統合は自然に合流する:

- **AI による自動タグ付け** — ノートの内容を AI が分析し、ローカルタグを自動提案
- **コレクション要約** — 蓄積されたコレクションを AI が要約・分析
- **自然言語検索** — 「先月の技術的な議論」等の曖昧なクエリを AI が FTS5 クエリに変換

## Phase 11: 管理者・開発者コンソール — 収益化の柱

NoteDeck を**サーバー管理者とフォーク開発者の司令塔**にする。
一般ユーザー向けの「構造的に対応しない領域」を、管理者ペルソナには積極的に取り込む。

### なぜ管理者・開発者をターゲットにするか

**1. 収益化の最有力ターゲット**

| ペルソナ | 月の支出感覚 | 寄付意欲 |
|---|---|---|
| 一般ユーザー | 無料が当然 | 低い |
| サーバー管理者 | サーバー代 ¥1,000〜¥10,000/月 | **高い（既に金を払う文化がある）** |
| フォーク開発者 | 開発環境に投資済み | **高い（時間を節約するツールに価値を感じる）** |

サーバー管理者は毎月サーバー代を払い、ドメイン代を払い、時間を投じている。
良いツールへの ¥500〜¥1,000/月 の支援は抵抗が低い。

**2. インフルエンサー効果**

管理者がNoteDeckを使う → サーバーの告知で「NoteDeckおすすめ」と言及 → そのサーバーのユーザーが流入。
フォーク開発者がNoteDeckに adapter を書く → そのフォークのユーザーがNoteDeckを使う。
**1人の管理者/開発者の獲得 = 数十〜数百人の一般ユーザーの獲得**。

**3. 既存資産との親和性**

- マルチサーバー同時接続（Phase 1）→ **複数サーバーを管理する管理者に直接価値**
- HTTP API + notecli（Phase 3-4）→ **運用自動化スクリプトとの連携**
- AiScript（Phase 3）→ **管理用カスタムスクリプト**
- ローカル DB（Phase 2）→ **モデレーションログの蓄積・検索**

### 11-1. モデレーションカラム

サーバー管理者の最頻業務。Web UI の管理画面より速く処理できることを目指す。

- [ ] **通報キューカラム** — 未処理の通報を一覧表示。ノート内容・通報理由・通報者をインライン表示。
  キーボードショートカットで承認/却下/ノート削除を即実行
- [ ] **ユーザーモデレーション** — ユーザープロフィールからワンクリックで凍結/サイレンス/ロール付与。
  フローティングウィンドウでモデレーション履歴も表示
- [ ] **モデレーションログカラム** — 管理操作の履歴を時系列で表示。
  ローカル DB にも蓄積し、FTS5 で検索可能（Web UI にはないオフライン検索）
- [ ] **横断モデレーション** — 複数サーバーの通報キューを統合表示。
  サーバーごとのテーマカラーで一目でどのサーバーの通報かわかる（NoteDeck の既存パターン）

### 11-2. サーバー監視ダッシュボード

常駐アプリだからこそのリアルタイム監視。ブラウザの管理画面を開きっぱなしにする必要がなくなる。

- [ ] **サーバーステータスカラム** — アクティブユーザー数、ノート流速（notes/min）、
  ドライブ使用量、ジョブキュー長をリアルタイム表示。
  admin/meta API + ストリーミングで常時更新
- [ ] **フェデレーション状況** — 接続中のリモートサーバー一覧、配送キューの状態、
  ブロック/サイレンス中のインスタンス管理
- [ ] **アラート通知** — ジョブキュー詰まり・ディスク使用量閾値超過・異常な通報増加等を
  デスクトップ通知。常駐アプリの強みを活かした早期検知
- [ ] **マルチサーバー比較ビュー** — 管理している複数サーバーの状態を並べて比較表示

### 11-3. フォーク開発者向けツール

NoteDeck を**フォーク開発のデバッグ・テスト環境**にする。
README で既に「フォーク対応の PR を歓迎」と書いている — これを技術的に支援する。

- [ ] **API エクスプローラー** — Misskey API をインタラクティブに叩けるカラム。
  エンドポイント選択 → パラメータ入力 → レスポンス表示。
  Swagger UI の NoteDeck 版。フォーク固有のエンドポイントもサーバー検出で表示
- [ ] **API レスポンス比較** — 同じエンドポイントを複数サーバー（本家 vs フォーク）に
  同時に叩き、レスポンスの差分をハイライト表示。
  adapter 開発時のフィールド差異確認に使う
- [ ] **ストリーミングデバッガー** — WebSocket ストリーミングのイベントをリアルタイム表示。
  イベントタイプ別フィルター・JSON プレビュー・ペイロード検索。
  フォーク固有イベントの挙動確認用
- [ ] **adapter テストランナー** — adapter のエンドポイントマッピングが正しく動作するかを
  接続中のサーバーに対してテスト実行

### 11-4. 管理者向け AiScript 連携

Phase 8-9 の AiScript エコシステム・自動化と合流する管理者特化機能。

- [ ] **管理用 AiScript API** — `Nd:admin.*` 名前空間で管理 API を AiScript に公開。
  `Nd:admin.reports`（通報一覧）、`Nd:admin.suspend`（凍結）等
- [ ] **モデレーション自動化テンプレート** — 「スパム自動検知」「特定パターンの通報自動処理」等の
  AiScript テンプレートをプラグインストア（Phase 8-2）で提供
- [ ] **運用レポート自動生成** — 日次/週次のサーバー運用レポートを AiScript + AI（Phase 6）で自動生成。
  新規ユーザー数・通報件数・TL 活性度等を要約

### 収益化モデル（検討）

NoteDeck はオープンソース（AGPL-3.0）を維持しつつ、持続可能な開発を支援する仕組みを検討する。

| モデル | 内容 | 検討状態 |
|---|---|---|
| GitHub Sponsors | 既存。管理者・開発者にリーチする導線を強化 | ✅ 実施中 |
| Sponsors 限定ベータ | 新 Phase の機能を Sponsors に先行提供 | 検討中 |
| 管理者向けプラグインパック | モデレーション・監視用の AiScript プラグイン集 | 検討中 |

**原則: コア機能は全て無料・オープンソース。**
管理者向け機能も最終的には全ユーザーに公開するが、Sponsors に早期アクセスを提供することで
支援のインセンティブを作る。

## Phase 12: Desktop Native Platform — Tauri だからできる差別化

Web UI / Flutter アプリには**アーキテクチャ的に不可能な**デスクトップネイティブ機能群。
Tauri の Rust バックエンド + OS レベルアクセスを最大限に活用する。

### なぜこれが重要か

NoteDeck は Tauri 製の唯一のデスクトップ Misskey クライアント。
この技術的ポジションから得られる差別化を体系的に実装し、
「デスクトップアプリである理由」を**原理的に追いつけない堀**として確立する。

### 12-1. ファイルシステム統合

Web はファイルシステムにアクセスできない。Flutter も制限付き。
Tauri は `std::fs` で自由にファイル操作できる。

- [ ] **ドラッグ＆ドロップ投稿** — デスクトップからファイルを投稿フォームにドロップ → 即アップロード。
  現在 `dragDropEnabled: false` を有効化し、DataTransfer でファイル受信
- [ ] **フォルダ監視 → 自動投稿** — `~/NoteDeck/auto-post/` にファイルを置く → 自動でアップロード＆投稿。
  スクリーンショットツールの保存先に設定すれば「撮った瞬間に投稿」
- [ ] **ノートのファイル保存** — ノート右クリック →「Markdown で保存」「画像として保存」。
  ローカルファイルに直接書き出し
- [ ] **添付ファイル一括ダウンロード** — 特定ユーザー / カラムの全添付ファイルを
  `~/Downloads/notedeck/{user}/` にバッチ保存
- [ ] **ffmpeg 連携** — 動画ファイルの自動トランスコード。
  サーバーのサイズ制限に合わせて圧縮してからアップロード

### 12-2. クリップボード深層統合

デスクトップアプリならではのクリップボード監視・変換機能。

- [ ] **クリップボード監視モード** — URL をコピー → NoteDeck が自動で OGP 取得 →
  「引用投稿しますか？」と通知 → ワンクリックで引用付き投稿
- [ ] **画像クリップボード検知** — PrintScreen / スクリーンショット撮影 → NoteDeck が検知 →
  「この画像を投稿しますか？」
- [ ] **リッチテキスト → MFM 変換** — Word やブラウザからリッチテキストをコピペ →
  太字・リンク等を MFM 記法に自動変換
- [ ] **ノートコピー形式選択** — ノート右クリック →「プレーンテキスト」「MFM」「Markdown」「HTML」でコピー

### 12-3. 外部プロセス連携

`tauri-plugin-process` が初期化済みだが未使用。Rust の `Command` で外部ツールと連携する。

- [ ] **画像編集連携** — 添付画像を右クリック →「GIMP で開く」「Photoshop で開く」。
  外部エディタで編集 → ファイル監視で変更検知 → 自動で再アップロード
- [ ] **ローカル LLM プロセス管理** — Ollama をサブプロセスとして起動・停止・状態監視。
  NoteDeck 起動時に自動起動、終了時に自動停止（Phase 6 と連携）
- [ ] **外部スクリプト実行** — AiScript プラグインから sh / Python スクリプトを呼び出し。
  `Nd:exec` API で安全なプロセス起動（ユーザー許可制）。
  「新着ノートを Slack に転送」「特定キーワードをスプレッドシートに記録」等
- [ ] **通知音カスタマイズ** — 任意の `.wav` / `.mp3` ファイルをローカルから指定。
  通知種別（リプライ・リアクション・フォロー）ごとに異なるサウンドを設定

### 12-4. ネイティブ通知の深い統合

`tauri-plugin-notification` は初期化済みだが活用が浅い。OS 通知を最大限に活用する。

- [ ] **アクション付き通知** — 通知ボタンから「いいね」「返信」を OS の通知パネルから直接操作。
  アプリに戻らずにリアクションできる
- [ ] **通知グルーピング** — 同一ユーザーからの連続通知をまとめて表示。
  「○○さんが 3 件リアクションしました」
- [ ] **Do Not Disturb 連携** — OS の集中モード / DND を検知 → 通知を自動抑制。
  DND 解除後に「集中モード中の通知まとめ」を表示
- [ ] **通知サウンドカスタマイズ** — 通知種別（リプライ・リアクション・フォロー・メンション）ごとに
  異なるサウンドを指定可能。ローカルの `.wav` ファイルも指定可能（12-3 と連携）

### 12-5. ローカルデータの主権

**思想: サーバーが消えても、あなたのデータはここにある。**
デスクトップアプリ + SQLite だからこそ実現できるデータ主権。

- [ ] **完全オフライン動作** — キャッシュ済みノートの閲覧・検索はネットワーク不要。
  飛行機の中でも過去の TL を読み返せる。サーバーが落ちても自分のデータは見られる
- [ ] **ワンファイルバックアップ** — `notedeck.db` を丸ごとコピーで完全バックアップ。
  サーバーが消滅しても自分の投稿・お気に入り・ローカルタグは残る
- [ ] **サーバー移転支援** — 旧サーバーのキャッシュを保持したまま新サーバーに移行。
  「あのノートなんだっけ」がローカル検索で見つかる
- [ ] **統計・分析ダッシュボード** — ローカル DB 上で SQL 集計 → 自分の投稿パターン分析。
  曜日別投稿数・リアクション傾向・よく使う絵文字・交流の多いユーザー等。
  Web UI には API がないため原理的に不可能な分析

### 12-6. localhost API の外部ツール連携

Phase 3 で実装済みの HTTP API（port 19820）を**外部ツール連携の正式プロトコル**に昇格。
API ドキュメントとサンプルコードの整備により、サードパーティ連携を促進する。

- [ ] **API ドキュメント公開** — OpenAPI (Swagger) 形式で API 仕様を公開。
  エンドポイント一覧・パラメータ・レスポンス例を自動生成
- [ ] **ランチャー連携サンプル** — Alfred / Raycast / Ulauncher 用のワークフローテンプレート。
  「検索」「投稿」「TL 表示」の基本操作をランチャーから実行
- [ ] **エディタ連携サンプル** — VS Code / Obsidian プラグインのサンプル実装。
  サイドバーに TL 表示、選択テキストを投稿、関連ノート検索
- [ ] **StreamDeck プラグイン** — 物理ボタンから投稿・TL 切替・ミュート・カラム操作。
  常駐 + HTTP API だからこそ実現できるハードウェア連携
- [ ] **Webhook 受信エンドポイント** — 外部サービスから NoteDeck にイベントを送信。
  GitHub の Push 通知 → NoteDeck のカラムに表示等

## Phase 間の依存関係

```
Phase 3 (Hackable)     Phase 5 (廃人モード)     Phase 6 (AI 統合)

  Phase 8      Phase 9     Phase 10                   │
  エコシステム   自動化       PKM ──────────────────── 合流
  Nd:* API     トリガー     タグ・コレクション        AI 自動タグ
  ストア       アクション    エクスポート             自然言語検索
       │          │            │
       ├──────────┴────────────┤
       │                      │
    Phase 11                Phase 7
    管理者・開発者         マルチウィンドウ
    コンソール             （独立して実装可能）
    Nd:admin.*
    モデレーション          Phase 12
    サーバー監視           Desktop Native Platform
                           （Phase 3 HTTP API を基盤に独立実装可能）
                           ファイルシステム・クリップボード
                           外部プロセス・通知・データ主権
                           API 外部ツール連携
```

**Phase 11 の依存関係:**
- Phase 8（Nd:* API）→ `Nd:admin.*` 管理 API の基盤
- Phase 9（自動化）→ モデレーション自動化テンプレート
- Phase 6（AI）→ 運用レポート自動生成
- Phase 1（マルチサーバー接続）→ 横断モデレーション・マルチサーバー監視

**Phase 12 の依存関係:**
- Phase 3（HTTP API）→ localhost API の外部ツール連携の基盤
- Phase 8（Nd:* API）→ `Nd:exec` 外部プロセス実行 API
- Phase 9（自動化）→ フォルダ監視・クリップボード監視の自動化トリガー
- 独立して実装可能な項目も多い（D&D 有効化、通知改善等は他 Phase に依存しない）
